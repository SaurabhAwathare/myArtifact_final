# ADR-0001: Repository Owns Recording Finalization

## Status
Approved

## Date
2026-06-25

## Context
Recording finalization involves updating multiple fields (duration, size, lifecycle) and potentially triggering background workers.

## Problem
If finalization logic is scattered in ViewModels or Services, it leads to duplication and risk of partial updates or state desync.

## Decision
All recording finalization logic must reside in `RecordingRepository.finalizeRecording()`. 

## Alternatives Considered
- **ViewModel-based orchestration**: Rejected because it requires repeating logic across different UI paths (e.g., normal stop vs. app crash recovery).
- **Service-based orchestration**: Rejected because services should coordinate operations but not implement persistence-heavy business rules.

## Consequences
- Single point of entry for ending a recording session.
- Easier to test finalization logic in isolation.
- Guarantees consistency between database state and file system state.
